## 📂 Post-Mortem Crash Dump Autopsy

### 📝 Description
When a C++ application crashes on your personal computer, you can simply attach a live debugger and watch the code line-by-line. However, when software is deployed on a remote client's production server and crashes at 3:00 AM, you cannot physically be there to investigate. Enterprise systems solve this using "Post-Mortem Debugging." When an application suffers a fatal crash, tools instantly take a frozen snapshot of the application's entire memory and CPU state the exact millisecond it dies. This snapshot is saved as a Crash Dump (`.dmp`) file. By transferring this file back to your workstation and loading it into a specialized debugger, you can perform a forensic autopsy—investigating the exact cause of death long after the process has vanished.

### 🎯 Objective
To intentionally trigger a fatal application crash, intercept the dying process mid-air to generate a Full Memory Dump, and perform a post-mortem autopsy to pinpoint the exact CPU register and assembly instruction responsible for the failure.

### 🧠 Major Underlying Concepts
* **Segmentation Fault / Access Violation (`0xC0000005`):** A fatal hardware-level error. This occurs when a program attempts to read or write data to a memory address it does not legally own—in this case, attempting to write to a "Null Pointer" (Memory Address `0x0`). 
* **Full Memory Dump (`.dmp`):** A forensic file generated by the operating system or a tool like ProcDump. It contains the exact state of the process's Virtual Address Space, active threads, and CPU registers at the moment of the crash.
* **Heuristic Engine Analysis:** Using the `!analyze -v` command in the Windows Debugger (WinDbg). This triggers Microsoft's automated forensic engine to sift through millions of bytes of raw memory to highlight the exact cause of the crash.
* **Instruction Pointer (IP):** A specific register inside the CPU that acts as a bookmark, pointing to the exact assembly instruction the computer was executing when the crash occurred.
* **The "Stripped Binary" Reality:** Analyzing an application without its Program Database (`.pdb`) symbol files. Because we did not generate a `.pdb` map to translate machine code back to human-readable C++ line numbers, this mimics a realistic scenario of reverse-engineering or incident response on a third-party application.

### 🛠️ Tool Usage & Physical Justification
* **Tools Used:** Sysinternals ProcDump (The Catcher) and Microsoft WinDbg (The Autopsy Table).
* **What to Look For:** Run the C++ program and attach ProcDump using the `-ma` (Memory Full) and `-e` (Exception) flags. Trigger the crash, allow ProcDump to generate the `.dmp` file, and load it into WinDbg. Run the `!analyze -v` command to sift through the data.
* **Physical Justification:** The console output explicitly proves the capture: `[18:44:58]Exception: C0000005.ACCESS_VIOLATION`, confirming ProcDump caught the fatal error and wrote the 9 MB `.dmp` file. Inside WinDbg, the autopsy physically reveals a zeroed-out CPU register (`0000000000000000`) and the specific assembly instruction attempting to write the hex value `2A` (decimal `42`), visually proving the exact root cause of the memory fault without ever looking at the original C++ source code.
